# План внедрения YandexGPT в проект Sector

## 1. Обзор

**Цель:** интегрировать YandexGPT (Яндекс.Облако) в backend и при необходимости в клиенты для генерации и улучшения текста в контексте мероприятий, чатов и Telegram-бота.

**Текущий стек:** Node.js (ESM), Express, MongoDB, Socket.IO, Telegram Bot API. Уже используется Yandex (геокодинг в tg-miniapp).

---

## 2. Предварительные требования

### 2.1 Регистрация в Яндекс.Облаке

- [ ] Создать аккаунт [Yandex Cloud](https://console.cloud.yandex.ru/)
- [ ] Создать каталог (folder)
- [ ] Включить сервис **YandexGPT** в каталоге
- [ ] Получить **OAuth-токен** (или использовать [IAM-токен](https://cloud.yandex.ru/docs/iam/operations/iam-token/create)) для сервисного аккаунта
- [ ] Сохранить **FOLDER_ID** (ID каталога)

### 2.2 Переменные окружения (backend)

Добавить в `.env` (и в настройки деплоя):

```env
# YandexGPT (опционально — если не заданы, фичи с GPT отключены)
YANDEX_OAUTH_TOKEN=...
YANDEX_FOLDER_ID=...
# или для сервисного аккаунта:
# YANDEX_IAM_TOKEN=...
# YANDEX_API_KEY=...  # если будет использоваться API Key
```

Документация по аутентификации: [Yandex Cloud — Authentication](https://cloud.yandex.ru/docs/iam/concepts/authorization/).

---

## 3. Архитектура интеграции

### 3.1 Слой доступа к API

| Компонент | Назначение |
|-----------|------------|
| `backend/src/services/yandexGptService.js` | Единая точка вызова YandexGPT: получение IAM-токена (если нужен), запрос к Text Generation API, обработка ошибок и лимитов |
| `backend/src/config/yandexGpt.js` | Конфиг: URL API, FOLDER_ID, включение/выключение фичи по флагу |

### 3.2 Варианты API YandexGPT

- **Синхронная генерация текста** — для коротких запросов (описания, подсказки).
- **Асинхронная генерация** — для длинных ответов (если понадобится позже).

Использовать официальный REST API:  
`POST https://llm.api.cloud.yandex.net/foundationModels/v1/completion` (актуальный endpoint уточнить в [документации](https://yandex.cloud/en/docs/yandexgpt/)).

## 4. Этапы внедрения

### Этап 1: Инфраструктура (backend)

1. **Конфиг и переменные**
   - Файл `backend/src/config/yandexGpt.js`: чтение `YANDEX_OAUTH_TOKEN`, `YANDEX_FOLDER_ID`, флаг `YANDEX_GPT_ENABLED`.
   - Если токен/папка не заданы — все вызовы YandexGPT возвращают «сервис недоступен» без падения.

2. **Сервис YandexGPT**
   - `backend/src/services/yandexGptService.js`:
     - `complete(prompt, options)` — один запрос к модели (system + user message или один prompt).
     - Ограничение длины ответа, таймаут запроса (например, 30 с).
     - Обработка ошибок (сеть, 429, 403) и логирование.
   - Не хранить токены в коде — только из `process.env` или конфига.

3. **Зависимости**
   - Оставить только `fetch` (Node 18+) или при необходимости добавить `axios` для запросов к API. Отдельный npm-пакет по желанию (например, обёртка над API).

### Этап 2: Первый сценарий — помощь в описании мероприятия

4. **Эндпоинт для генерации описания**
   - Роут: `POST /events/suggest-description` (или `POST /ai/suggest-event-description`).
   - Тело: `{ "title": "...", "place": "..." }` (опционально: черновик `draft`).
   - Логика: собрать промпт вида «По заголовку и месту мероприятия сгенерируй краткое описание 1–2 абзаца для анонса», вызвать `yandexGptService.complete()`, вернуть `{ description: "..." }`.
   - Права: только авторизованный пользователь (middleware `authMiddleware`).

5. **Интеграция в создание/редактирование события**
   - В `eventService.js` не менять — генерация только по явному запросу с фронта.
   - Фронт (sector-web / tg-miniapp): кнопка «Подсказать описание» в форме создания/редактирования события, запрос на новый эндпоинт, подстановка результата в поле описания (пользователь может править).

### Этап 3: Дополнительные сценарии (по приоритету)

6. **Улучшение текста**
   - Эндпоинт: `POST /ai/improve-text` с `{ "text": "..." }` — возврат отредактированного/улучшенного текста (коррекция, стиль). Использовать в формах событий и при необходимости в чате.

7. **Краткое резюме чата мероприятия**
   - Эндпоинт: `POST /chats/:chatId/summary` (или `/events/:eventId/chat-summary`).
   - Для организатора: последние N сообщений чата → промпт «Сделай краткое резюме обсуждения» → ответ в 2–3 предложения. Кэширование на короткое время (например, 5–10 мин), чтобы не дергать API при каждом открытии.

8. **Telegram-бот**
   - Обработка произвольных текстовых сообщений (не только команд): при неизвестной команде можно отправить текст в YandexGPT с контекстом «Ты помощник бота SEKTOR. Отвечай кратко о мероприятиях и приложении» и вернуть ответ пользователю.
   - Либо отдельная команда `/ask <вопрос>` для вопросов к боту.

9. **Напоминания о мероприятиях**
   - В `reminderService.js` сейчас шаблонный текст. Опционально: один раз при формировании напоминания вызывать YandexGPT для короткой персонализированной строки (например, «Не забудь: завтра встреча в коворкинге»). Учитывать лимиты и задержки — делать вызов асинхронно и с таймаутом, при ошибке использовать текущий шаблон.

### Этап 4: Безопасность и лимиты

10. **Лимиты и квоты**
    - Ограничить длину промпта и ответа в `yandexGptService.js` (по документации API).
    - Rate limit на уровне приложения: например, не более N запросов в минуту на пользователя для `/ai/*` (express-rate-limit или аналог).
    - Учёт квот Яндекс.Облака (лимиты на количество токенов в день/месяц).

11. **Безопасность**
    - Не передавать в промпты персональные данные без необходимости; не логировать полные ответы с персональными данными.
    - Эндпоинты `/ai/*` только для авторизованных пользователей.
    - В Telegram-боте не принимать от пользователя произвольные длинные запросы без ограничения длины.

12. **Мониторинг**
    - Логировать факт вызова YandexGPT, время ответа, код ошибки (без тел запросов/ответов в лог при необходимости соответствия политике).
    - Метрики (если есть): количество запросов, ошибки, latency.

## 5. Структура файлов (итог)

```
backend/
  src/
    config/
      yandexGpt.js          # конфиг, флаг включения
    services/
      yandexGptService.js   # вызов API YandexGPT
    routes/
      aiRoutes.js           # POST /ai/suggest-description, /ai/improve-text и т.д.
    controllers/
      aiController.js       # обработчики для AI-эндпоинтов
```

Подключение в `app.js`: `app.use("/ai", aiRoutes)` (с `authMiddleware` на роутере).

---

## 6. Тестирование

- **Unit:** мок `yandexGptService` в тестах `aiController` и сервисов, проверка формата ответа и обработки «сервис выключен».
- **Интеграция:** один-два теста с реальным API в CI не обязательны (зависит от секретов); достаточно ручной проверки с реальным токеном в dev.
- **E2E:** сценарий «создать событие → нажать „Подсказать описание“ → проверить, что поле заполнилось».

---

## 7. Риски и митигация

| Риск | Митигация |
|------|-----------|
| Недоступность API / таймаут | Все вызовы с таймаутом; при ошибке возвращать понятное сообщение и не ломать основной флоу (например, форма события остаётся без подсказки). |
| Превышение квот | Rate limit на пользователя; мониторинг использования; при 429 — ответ «попробуйте позже». |
| Стоимость | Оценить цену за токен в Yandex Cloud; при необходимости ограничить сценарии или частоту вызовов. |
| Конфиденциальность | Не отправлять в промпты чувствительные данные; в политике/README указать использование YandexGPT для подсказок. |

## 8. Порядок работ (чек-лист)

- [ ] **Неделя 1:** Регистрация в Yandex Cloud, получение токена и FOLDER_ID; конфиг `yandexGpt.js` и сервис `yandexGptService.js`; один тестовый эндпоинт `GET /ai/health` (проверка доступности без генерации).
- [ ] **Неделя 2:** Эндпоинт `POST /ai/suggest-event-description`, интеграция в форму создания/редактирования события (sector-web и/или tg-miniapp).
- [ ] **Неделя 3:** Эндпоинт улучшения текста; при необходимости — резюме чата; rate limit и обработка ошибок.
- [ ] **Неделя 4:** Опционально — Telegram-бот (ответы на вопросы); опционально — улучшение текста напоминаний; документация и мониторинг.

После выполнения плана можно расширять сценарии (например, генерация анонсов для соцсетей, теги мероприятий по описанию и т.д.).
